Product Concept

Why “FirstBeam”?

When the whole task feels dark, the user does not need a perfect plan first. They need the first beam of clarity: one small action that feels possible enough to begin.

01

Create a beacon

The user enters one stressful task, and FirstBeam turns it into a small set of approachable subtasks.

02

Make it smaller

With a Gemini key, shrink can generate new micro-steps; without one, it falls back locally.

03

Move into focus

The product connects planning with action through a timer-centered focus mode.

04

Relight later

Archived or overdue tasks can return with a new deadline, without shaming the user.

05

Analyze habits

The profile area can turn completed timing history into guidance for future breakdowns.

06

Import / export records

Users can download a local JSON backup and restore previous records from that file.

Architecture

Local-first frontend, agent-driven backend.

The current repo contains both a mobile-first app experience and a real Google ADK multi-agent backend. The demoed frontend is hybrid: persistent state stays local, while AI generation can use the user’s own Gemini key.

Why agents? A single to-do list only stores work. FirstBeam needs different behaviors for different moments: planning a new task, shrinking a step that still feels too big, and reflecting after progress.

Data Safety

User records stay on the user's laptop or phone.

FirstBeam’s strongest accurate privacy claim is local-first persistence: previous tasks, stats, and habit profile data are stored in the browser’s localStorage, not in a project-owned server database.

Privacy

No app database for task records

The repository does not include server-side persistence for the user's task history.

AI nuance

Model calls are optional

With a Gemini key, intake, shrink, and habit analysis can send relevant text for that request. Persistent records remain local.

Security

No hardcoded API secrets

The app uses a user-supplied API key model instead of committing private keys into the codebase.
